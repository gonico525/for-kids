# こどもひろば コンセプト

このドキュメントは「こどもひろば」が **何のために・誰のために・どういう思想で** 作られているかをまとめたもの。
改修や新アプリ追加のときに「この変更はこのアプリ群らしいか?」を判断する指針にする。
実装の手順・API は [CLAUDE.md](../CLAUDE.md) と [docs/reference/](reference/) を参照(このドキュメントは WHY、あちらは HOW)。

## 一言でいうと

**幼児期(4〜6歳)の能力開発に効果のエビデンスがある活動を、まだ文字が読めない子どもがひとりで安心して遊べる形にした、おうちのタブレット/スマホ向けアプリ集。**

## 対象ユーザー

ユーザーは2種類いる。

- **子ども(4〜6歳)** — メインのプレイヤー。ひらがなも数字もまだ読めない・読み始めの段階。
  端末は保護者のスマホや家庭用タブレット(古めの Fire HD 10 など性能・絵文字フォントが古い端末を含む)。
- **保護者** — 端末を渡す人・設定する人。「遊ばせすぎない」「変な広告や課金がない」「学びにつながる」ことを求めている。
  設定 UI(保護者ゲート)だけは大人向けで、漢字・文章を使ってよい。

## 体験の柱(デザイン原則)

### 1. エビデンスのある活動だけを取り入れる

**このアプリ群のいちばんの選定基準。** あそび・べんきょうの区分に関わらず、すべてのアプリは
「なんとなく楽しそう」「子ども向けアプリでよくあるもの」ではなく、
**幼児期の能力開発として有効性が研究で裏付けられている活動**をもとにしている。

各アプリが狙っている能力と、その活動のエビデンスの現状(2026-07 時点の文献調査。◎=裏付けが強い / ○=裏付けあり / △=弱い・間接的):

| アプリ | 狙っている能力 | エビデンス |
|---|---|---|
| 🐹 ぴょこっとタッチ(あそび) | 抑制制御(押す/押さないの判断)・選択的注意・反応速度 | △ 訓練した課題自体と近い課題は上達するが、日常の自制心への般化(far transfer)の裏付けは弱く一貫しない |
| 🧍 だるまさんがころんだ(あそび) | 自己制御(「動く/止まる」の切り替え)+基礎的な動きの経験(移動系・バランス系、幼児期運動指針の「多様な動き」) | ◎ ストップ&ゴー型の集団遊び(Red Light, Purple Light!)の RCT で自己制御の向上が示されている。幼児期の運動遊び介入も運動能力向上のメタ分析あり(エビデンスの質は低〜中) |
| 🚃 ぴたっとストップ(あそび) | タイミングの予測(coincidence-anticipation timing)・目と手の協働 | △ これらの能力が幼児期に発達すること自体は確立しているが、画面上のタイミング課題の訓練効果を4〜6歳で直接調べた研究は見つからず |
| 🧮 ヌマーカステン(べんきょう) | 数感覚(数の合成・分解、量の直観的把握=サビタイジング) | ◎ サビタイジング・合成分解を扱う数感覚介入の RCT で算数力の向上が示されている。教具レーケンレック自体の効果検証は少ないが、狙う構成概念の裏付けは強い |
| 🔮 つぎはな〜に?(べんきょう) | パターン認識(初期の数学的思考の基礎) | ◎ 就学前のパターン理解が後年(小5〜6まで)の算数力を独自に予測し、パターン訓練の介入研究でも算数力向上が示されている。6アプリ中で最も裏付けが強い |
| ➡️ やじるしめいろ(べんきょう) | 順序立て・プログラミング的思考・空間認識 | ○ Bee-Bot 型の逐次プログラミング介入の RCT で実行機能・順序立ての向上が報告されている。ただし幼児プログラミング研究の蓄積はまだ浅い |

エビデンスが △ のアプリを外す必要はない(害があるわけではなく、楽しさ・反応速度など別の価値もある)が、
**△ のアプリを「能力開発に効く」と過信して増やさない**こと。新規追加はできるだけ ◎○ 相当の活動から選ぶ。
なお現状の △ は2つともチケット制の「あそび」側にあり、接する時間が自然に区切られる配置になっている。
無制限の「べんきょう」側に置くアプリは ◎○ 相当の裏付けを必須とする。

主な出典:

- ストップ&ゴー遊びと自己制御: [Tominey & McClelland (2011) "Red Light, Purple Light" RCT](https://www.tandfonline.com/doi/abs/10.1080/10409289.2011.574258)、
  [Blueprints 認定の効果評価](https://www.blueprintsprograms.org/programs/1500999999/red-light-purple-light-a-self-regulation-intervention/print/)、
  [2019年の追試(自己制御の有意な向上は再現されず)](https://pmc.ncbi.nlm.nih.gov/articles/PMC6817624/)
- 幼児の運動遊び: [幼児期運動指針(スポーツ庁)](https://www.mext.go.jp/sports/b_menu/sports/mcatetop03/list/1396909.htm)、
  [保育施設での運動介入のメタ分析 (Wick et al. 2017)](https://link.springer.com/article/10.1007/s40279-017-0723-1)
- 抑制制御のコンピュータ訓練(転移が限定的): [Liu et al. (2015)](https://www.nature.com/articles/srep14200)
- 数感覚介入の RCT: [Jordan et al. "Building Kindergartners' Number Sense"](https://www.researchgate.net/publication/239840203_Building_Kindergartners'_Number_Sense_A_Randomized_Controlled_Study)
- パターン理解と算数力: [就学前のパターン力が後の算数力を予測 (Rittle-Johnson ら)](https://www.sciencedirect.com/science/article/abs/pii/S0022096520304197)、
  [レビュー論文](https://pmc.ncbi.nlm.nih.gov/articles/PMC9914987/)
- 幼児の逐次プログラミングと実行機能: [Di Lieto et al. (2017) Bee-Bot RCT](https://www.sciencedirect.com/science/article/abs/pii/S0747563217300195)、
  [ScratchJr 研究レビュー](https://pmc.ncbi.nlm.nih.gov/articles/PMC11109739/)

**新しいアプリ・活動を追加するときは、実装より先に次の3点を確認する**(アイデアの確からしさの検証):

1. **時期が適切か** — その能力の発達において 4〜6歳が適時か。早すぎ/遅すぎないか
2. **エビデンスがあるか** — その活動が目的の能力の発達に有効だという研究的裏付けがあるか。
   「デジタルゲーム化しても効果が残るか」まで考える(実物・身体を使う方が良い活動もある)
3. **より良い代替がないか** — 同じ能力を狙うなら、より効果が確かな・この年齢に合った活動が他にないか比較する

この確認で分が悪ければ、楽しそうでも採用しない。

### 2. 文字が読めなくても遊べる

対象年齢は文字が読めない。**絵文字・色・音・アニメーションで伝える**のが大原則。

- ボタンやフィードバックは絵文字+色(緑=OK/赤=NG など)で意味が分かるようにする
- テキストは補助。書く場合もひらがな+分かち書き(「きょうは あと」)で、読めなくても成立するようにする
- 効果音は「正解/不正解/開始/終了」が音だけで区別できるように設計する(`KidsSound.tone()` で合成)
- 例外は保護者ゲートまわりのみ(大人向けなので通常の日本語でよい)

### 3. 遊ばせすぎない — 🎈チケットと「あそび/べんきょう」の非対称

保護者が安心して渡せることも重要。そのための仕組みが **カテゴリの非対称設計**:

- **🎈 あそび(`games/`)** — 1日の回数制限あり。全ゲーム合算で🎈チケット(既定3枚、保護者が1〜5枚に設定可)を消費する。
  **毎ラウンド開始時に必ず `KidsTickets.useOne()` でゲートし**、チケットが尽きたら「おやすみ画面」を出して終わりにする。
- **✏️ べんきょう(`study/`)** — **無制限**。チケットを一切消費しない。

注意: この区分は「あそび=ただの娯楽」という意味では**ない**。どちらのカテゴリも原則1のとおり
能力開発のエビデンスがある活動で、あそび側も抑制制御や自己制御といった実行機能を狙っている。
非対称なのは熱中のしやすさへの対処で、反射・スコア中心で夢中になりやすいものはチケットで
回数を区切り、遊びたい気持ちが残っているときに無制限の「べんきょう」へ自然に流れるようにしている。

新アプリを追加するときは「これはあそびか、べんきょうか」を最初に決めること。境界が曖昧な場合、
反射・スコア中心で熱中しやすいならあそび、じっくり考える練習ならべんきょう。

- 設定変更(チケット上限・リセット)は**保護者ゲート**(6〜9 × 6〜9 の掛け算クイズ)の先にだけ置く。
  子どもが偶然突破できない程度でよく、セキュリティ機能ではない。
- 「おしまい 💤」は罰ではなく、おやすみの挨拶として優しく演出する。

### 4. オフライン完結・外部依存ゼロ

- **ネットワークに一切依存しない**: CDN のフォント/スクリプト禁止、外部 API 禁止、音声・画像素材も持たない
  (音は WebAudio 合成、絵は絵文字とCSS)。一度キャッシュされれば機内モードでも全機能が動く(PWA + Service Worker)
- **ビルドシステム・フレームワーク・パッケージマネージャを導入しない**: 各アプリは自己完結の1枚 HTML。
  「ブラウザで開けば動く」「1ファイル読めば全部わかる」を維持する
- 通信しない・アカウントがない・課金がない・広告がないことは、保護者への安心の約束でもある。
  データは端末内 `localStorage` のみ(キーは [docs/reference/storage-keys.md](reference/storage-keys.md) に登録)

### 5. 古い端末・多様な環境で動く

家庭の「お下がり端末」で動くことを重視する。

- 絵文字は古い世代(Unicode 6 前後)を優先。新しい絵文字が必要なら `KidsEmoji.fix()` でフォールバックを宣言する
  (Fire HD 10 で豆腐になる事故が実際にあった)
- レイアウトはスマホ縦持ちファーストで、4パターン(縦持ち / タブレット / 横2カラム / 低い横画面)を必ず確認する
- `prefers-reduced-motion: reduce` のフォールバックを全アニメーションに用意する

### 6. 学びは「できた!」の積み重ね

べんきょうアプリの進行設計の共通思想:

- **失敗を責めない**: 不正解でもやり直せる。減点・失敗演出は最小限に、正解の演出を厚くする
- **レベルは自動で上がる**: レベル制アプリは ⭐ を5個ためると自動レベルアップ(`KidsProgress`)。
  子どもが自分で「むずかしいのにする」必要がないようにしつつ、手動切替も残す
- **短いループ**: 1問・1ラウンドが数十秒で完結し、達成感が細かく得られるようにする

## やらないこと(非ゴール)

「便利そう」「楽しそう」でも以下はこのアプリ群の思想に反するので入れない:

- **エビデンスのない「楽しいだけ」のアプリ・要素** — よくある子ども向けゲームの真似や思いつきの
  ミニゲーム。原則1の3点確認(時期・エビデンス・より良い代替)を通らないものは追加しない
- **オンライン要素全般** — ランキング、共有、通知、アカウント、外部トラッキング
- **広告・課金・外部リンク** — 子どもが触る画面から外に出る導線を作らない
- **文字前提の UI** — 説明文を読まないと遊べない設計(読み上げ等の補助で解決する)
- **ビルドシステム・フレームワーク・npm 依存の導入**
- **ゲームロジックや CSS テーマの共通化のしすぎ** — 色・世界観はアプリごとに意図的に変えている。
  共有するのはインフラ(チケット・音・起動処理・進行)まで

## 変更時のチェックリスト

改修・追加のたびにこの問いに YES と言えるか確認する:

**新しいアプリ・活動を足すとき(実装前に):**

- [ ] その能力の発達に 4〜6歳という時期が適切か
- [ ] その活動に能力開発のエビデンスがあるか(デジタル化しても効果が残る見込みか)
- [ ] 目的の能力に照らして、より良い活動が他にないか比較したか
- [ ] 狙う能力を原則1の一覧表に追記したか

**すべての変更で:**

- [ ] 文字が読めない子が、見て・聞いて操作を理解できるか
- [ ] オフライン(機内モード)で完全に動くか。外部リソースを増やしていないか
- [ ] あそびならラウンド開始が `KidsTickets.useOne()` でゲートされているか。べんきょうならチケットに触れていないか
- [ ] 新しい絵文字を使うなら `KidsEmoji.fix()` のフォールバックがあるか
- [ ] 4レイアウト(縦持ち / タブレット / 横2カラム / 低い横画面)で確認したか
- [ ] アニメーションに `prefers-reduced-motion` の配慮があるか
- [ ] `localStorage` キーを増やしたら [storage-keys.md](reference/storage-keys.md) に登録したか
- [ ] `sw.js` の `ASSETS` / `CACHE_NAME` を更新したか(`python3 tools/check_sw.py` で確認)
